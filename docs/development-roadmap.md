# 开发路线与验收标准

> 最后更新：2026-07-17
> 用法：任何新开发窗口先读 `AGENTS.md` 和 `docs/handoff.md`，只执行本文档中“当前阶段”的一个小任务。完成后更新状态和交接，不跨阶段堆功能。

## 1. 状态说明

- `[完成]`：代码、自动化测试和必要的真实验证均通过。
- `[进行中]`：当前允许开发的阶段。
- `[待开始]`：已设计，但不能写成已实现功能。
- `[可选]`：不影响简历主项目完成度。

## 2. 总体顺序

```text
阶段 1-4：MVP（已完成）
-> 阶段 5：登录、用户隔离与管理员管理（已完成）
   -> 5.5 Vue 登录体验
   -> 5.6 管理员与系统文档管理
-> 阶段 6：Redis 接口保护
-> 阶段 7：RAG 评估与检索优化
-> 阶段 8：可观测性
-> 阶段 9：受控资料整理 Agent
-> 阶段 10：Docker 与云端发布
```

一次只完成一个可验证的小任务。禁止同时改认证、Redis、RAG 和 Agent。

对应发布里程碑：

- `RAG v1.1`：阶段 6 完成，具备限流、并发保护和幂等性。
- `RAG v1.2`：阶段 7 完成，检索效果可以用固定评估集证明。
- `RAG v1.3`：阶段 8 完成，可以解释请求耗时、费用和失败原因。
- `Agent v2.0`：阶段 9 完成，具备独立、受控、可审计的资料整理 Agent。
- `Cloud v2.1`：阶段 10 完成，可从空服务器重复部署并安全保留数据。

这里的“对标高星项目”指学习清晰模块、运行状态、评估、审计和故障边界，不是复制 Dify、RAGFlow、Coze、Langflow 或 OpenHands 的全部平台功能。

## 4. 阶段五：登录和用户隔离 `[完成]`

### 4.1 固定产品决策

- 邮箱和密码注册登录。
- 每个账号只看到自己的历史会话。
- 当前旧会话是测试数据，迁移时备份后清空。
- 知识库全部公共，登录用户都可检索。
- 新上传文档也公开，但只有上传者可以删除。
- 系统预置文档不能由普通用户删除。
- 第一版只做短期 Bearer JWT。
- 基础登录版本不做邮箱验证码、找回密码和 Refresh Token。
- 完成 Vue 登录后增加管理员角色；当前普通用户上传公共资料的规则保持不变。

### 4.2 建议拆分任务

#### 任务 5.1：迁移基础和用户模型 `[完成]`

范围：

1. 引入 Alembic，并为当前三张会话表生成可复现基线。
2. 新建 `modules/auth` 的 model、schema、repository、service 骨架。
3. 创建 `users` 表，邮箱唯一，密码只保存哈希。
4. 增加迁移和模型测试。
5. 不修改现有页面，不接 JWT，不清空会话。

验收：

- 空数据库可通过 Alembic 升级到最新结构。
- 已有数据库可执行升级且原三张表数据不丢失。
- 重复邮箱被数据库和服务层正确拦截。
- 数据库和日志中都没有明文密码。
- 后端原 31 项测试仍通过，并新增认证模型与迁移测试，当前共 37 项。

#### 任务 5.2：注册、登录和当前用户接口 `[完成]`

范围：

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`
- 密码哈希、JWT 签发和认证依赖。

验收：

- 正确注册登录可获得 Token。
- 错误邮箱/密码只返回统一安全提示。
- 过期、伪造、缺失 Token 均返回 401。
- API Key、JWT 密钥和密码不进入 Git 或日志。

#### 任务 5.3：会话归属和旧数据迁移 `[完成]`

范围：

- `conversations` 增加 `user_id`。
- 迁移前导出旧测试会话，随后按用户决策清空。
- 会话所有查询和操作绑定当前用户。

验收：

- A 用户看不到、读不到、改不了、删不了 B 用户会话。
- 猜测其他会话 UUID 仍返回 404，不泄漏资源存在。
- SSE 普通完成、失败、中止状态仍正确落库。

#### 任务 5.4：文档登记迁移和公共权限 `[完成]`

范围：

- 建立 `documents` 表，替代 `documents.json` 为主登记源。
- 现有文档迁移为系统公共文档。
- 新上传记录 `uploader_id`。
- 公共检索、上传者删除、系统文档保护。

验收：

- 迁移前后文档数量和 Chroma 片段数量一致。
- 不重新调用 Embedding，不产生重复片段。
- 任意登录用户可检索公共文档。
- 非上传者不能删除，系统文档不能由普通用户删除。
- 删除仍验证 MySQL、文件、Chroma 三处一致性。

#### 任务 5.5：Vue 登录体验 `[完成]`

范围：

- 注册和登录页。
- Token 管理、当前用户状态和退出登录。
- 路由守卫和 401 统一处理。
- 登录后加载当前用户会话。

验收：

- 未登录访问问答/知识库会跳到登录页。
- 登录成功返回原目标页面。
- 刷新后在 Token 有效期内保持登录。
- 退出后清空本地凭据和当前会话状态。
- 浏览器控制台无错误，前端测试和构建通过。

#### 任务 5.6：管理员与系统文档管理 `[完成]`

范围：

- 通过 Alembic 为 `users` 增加 `role`，第一版只允许 `user` 和 `admin`。
- 增加统一的 `require_admin` 认证依赖，管理员权限必须由后端校验。
- JWT 仍只保存用户 ID，角色以数据库为准；`/auth/me` 返回角色供前端展示。
- 提供系统文档新增、删除和整份替换接口。
- 普通文档与系统文档共用一个文档生命周期组件，禁止复制解析、切片、向量化、快照和回滚逻辑。
- 增加独立的管理员知识库页面，不把管理逻辑塞进普通问答页面。
- 修复 `backend/scripts/import_documents.py`，让它显式使用管理员身份或受控维护服务，不绕过正常文档流程。
- 普通登录用户仍可上传公开资料并删除自己的上传；管理员额外负责系统文档。

验收：

- 新注册用户默认是 `user`，不能伪造或自行提升为 `admin`。
- 管理员初始化采用受控命令或安全配置，不提供公开的管理员注册接口。
- 普通用户调用管理员接口统一返回 403。
- 修改 localStorage、JWT载荷或请求参数都不能获得管理员权限；用户降权后旧Token立即失去管理员能力。
- 管理员新增系统文档后，所有登录用户都能检索且普通用户不能删除。
- 系统文档替换采用新版本先就绪、旧版本后清理的可补偿流程，不宣称跨存储原子事务。
- 新增、删除或替换任一步失败时，三处数据恢复到操作前状态。
- 同一文档的并发替换被拒绝或串行执行；不能出现两个管理员互相覆盖。
- 批量导入脚本可重复运行，重复内容跳过，且不绕过哈希去重、切片和向量登记。
- 管理页面完成组件测试和真实浏览器验收。
- 普通注册登录、会话隔离、普通上传删除、RAG检索和SSE回归全部通过，任务5.6不得改变这些接口契约。

### 4.3 阶段完成标准

- 两个账号的数据隔离测试通过。
- 公共文档检索和删除权限测试通过。
- 普通用户与管理员的系统文档权限测试通过。
- 现有 RAG、SSE、刷新恢复和停止生成无回归。
- README 和演示步骤更新。

## 5. 阶段六：Redis 接口保护 `[完成]`

### 建议拆分任务

1. **任务 6.1 `[完成]`**：增加 Redis 配置、连接适配器、健康状态和可测试的故障策略，不接入业务限流或锁。
2. **任务 6.2a `[完成]`**：只为注册和登录增加按 IP 的有界限流及统一 429 响应。
3. **任务 6.2b `[完成]`**：只为普通聊天和会话聊天增加按用户的频率保护，不改上传和生成锁。
4. **任务 6.2c `[完成]`**：只为上传增加按用户的频率与并发保护，不改文档生命周期。
5. **任务 6.3a `[完成]`**：增加同一用户同一会话的生成锁，验证所有权令牌、TTL、中止和异常释放。
6. **任务 6.3b `[完成]`**：增加客户端请求 ID 和幂等记录，防止网络重试重复创建消息或重复调用模型。
7. **任务 6.4a `[完成]`**：完成 Redis 保护功能的故障矩阵、结构化日志与可观测状态，不改页面。
8. **任务 6.4b `[完成]`**：前端稳定提示、可取消的后端模型流、真实 Redis/HTTP/页面复测均通过，`RAG v1.1` 已冻结。

任务 6.1 验收：

- Redis 地址和超时只从环境配置读取，缺失配置不会导致与 Redis 无关的接口无法启动。
- 连接、ping、关闭和健康状态由 Infrastructure 适配器集中封装，路由和业务 Service 不直接创建 Redis 客户端。
- 未配置、连接失败和运行中断都有明确状态、有限超时、稳定日志与自动化测试。
- 健康接口能区分“应用可运行但 Redis 未启用/降级”和“Redis 正常”，不输出连接密钥。
- 后端完整回归通过，且代码中尚未出现限流计数、会话锁或幂等键业务逻辑。

任务 6.2a 验收：

- 注册和登录分别使用明确的按 IP 窗口与上限，计数键有 TTL，不会无限增长。
- 只有受信代理配置启用时才读取转发地址，默认使用直接连接地址，避免伪造请求头绕过限流。
- 超限稳定返回 429、统一错误码、`request_id` 和合理的 `Retry-After`，不泄漏账号是否存在。
- Redis 不可用时采用明确的安全降级并记录告警，不能无声放开无限登录尝试。
- 注册登录原有成功、失败和安全提示测试保持通过；聊天、上传和生成流程没有改动。

任务 6.2b 验收：

- 普通聊天、会话非流式问答和 SSE 问答都在调用检索或模型前按当前用户消费额度，不遗漏任一入口。
- 用户键使用稳定摘要，不包含 Token、邮箱、问题正文或其他敏感信息；不同用户额度互不影响。
- 非流式和 SSE 接口均能在响应开始前返回统一 429、`request_id` 与 `Retry-After`，不把限流错误伪装成成功流事件。
- Redis 不可用时继续使用有容量和自动过期上限的本机兜底并记录降级，多实例限制需在文档中如实说明。
- 原聊天成功、失败、停止生成和历史恢复测试保持通过；上传、生成锁、幂等、Prompt 和检索算法没有改动。

任务 6.2c 验收：

- 上传在解析、Embedding、文件和 Chroma 写入前按当前用户检查频率，并限制同一用户同时进行的上传数量。
- 频率键和并发占位键使用独立命名空间、稳定用户摘要和有限 TTL，不包含 Token、邮箱、文件名或文档正文。
- 超限或并发占满返回稳定 429、`request_id` 与合理的 `Retry-After`；拒绝请求不产生临时文件、向量或数据库登记。
- Redis 故障策略明确且有测试，不能静默放开无限昂贵上传；并发占位的释放必须覆盖成功、解析失败、Embedding 失败和客户端异常。
- 原上传去重、权限、跨 MySQL/文件/Chroma 补偿测试保持通过；不修改共享文档生命周期、聊天、生成锁、Prompt 或检索算法。

任务 6.3a 验收：

- 会话非流式和 SSE 问答在保存消息、检索和模型调用前，以“当前用户 + 会话”为键获取同一把生成锁；普通无会话问答暂不接入。
- 锁使用随机所有权令牌和有限 TTL，释放时原子比较令牌，旧请求不能删除后来请求获得的锁。
- 已占用时返回稳定冲突错误与 `request_id`；Redis 未配置、连接失败或锁状态不明确时返回明确临时不可用，不能继续产生模型费用。
- 非流式成功/失败，以及 SSE 完成、模型失败、主动停止、客户端断开和生成器关闭都释放锁；进程崩溃由 TTL 最终回收。
- 原会话归属、消息状态、历史恢复、聊天限流和上传保护测试保持通过；不同时实现客户端请求 ID、幂等记录或修改 RAG 行为。

任务 6.3b 验收：

- 会话非流式和 SSE 问答接受受约束的客户端请求 ID；幂等键绑定当前用户、接口和请求 ID，不包含 Token、问题正文或邮箱。
- 同一个幂等请求重复到达时不重复保存消息、检索或调用模型，并能返回原有资源标识或稳定结果摘要；不缓存完整医疗回答。
- 首次请求进行中、成功、业务失败、客户端断开以及幂等存储故障都有明确状态、有限有效期和自动化测试，不留下永久卡死记录。
- 幂等检查与会话生成锁的顺序明确，既不能绕过生成锁，也不能把正常网络重试错误地当作另一个新生成任务。
- 原会话归属、消息状态、历史恢复、聊天限流、生成锁和上传保护测试保持通过；不修改 RAG Prompt、检索算法或普通无会话问答。

任务 6.4a 验收：

- 用一张可测试的故障矩阵覆盖 Redis 未配置、连接失败、命令失败、恢复、锁状态不明确和幂等结果登记失败，明确每种功能是本机兜底还是 fail-closed。
- 结构化日志和可观测状态只记录功能、结果、降级原因类型和必要计数，不记录 Redis URL、Token、邮箱、问题、回答或文档正文。
- 健康或独立状态接口能区分限流兜底、生成锁不可用和幂等不可用，不把应用仍存活误报为整体宕机。
- 自动化测试证明日志不会重复刷屏、故障恢复可见且不会因为观测代码改变原 429/409/503 行为。
- 不安装 Redis、不修改 Vue 页面、不调用模型，也不修改 RAG、上传生命周期、Agent 或 Docker。

任务 6.4b 验收：

- 前端对 429、生成冲突 409、幂等进行中/复用冲突和 Redis 临时不可用 503 给出稳定、可理解且不过度技术化的提示。
- 在用户确认的可用 Redis 环境中完成一次真实 HTTP 和浏览器验收，证明首次生成、重复请求恢复、并发冲突与停止生成符合契约。
- 停止生成不能只依赖浏览器断开：前端发送带本次请求标识的停止请求，后端保存 `stopped`、清理幂等状态并释放生成锁后，页面才恢复下一次发送。
- 模型下一块数据长时间未返回时，停止仍须取消本地异步读取并关闭 DashScope HTTP 流；自动化阻塞测试不能只验证数据块之间的取消标记。
- 冻结阶段六配置、错误码、文档和演示步骤，完整回归通过后标记 `RAG v1.1`。

后续任务共同边界：

- `RateLimitService`、`DistributedLockPort` 和 Redis Adapter 分层，Router/业务 Service 不执行 Redis 命令。
- 限流、锁和幂等使用独立键空间与 TTL，不在键中暴露邮箱、Token 或问题正文。
- Redis 只保存可重建的短期状态，不保存用户、会话、文档或 Agent 运行的唯一副本。
- 锁释放必须验证所有权令牌，Redis 状态不明确时不得继续产生昂贵模型调用。
- 每小步只修改对应接口，不能顺手缓存回答、改 RAG Prompt 或引入消息队列。

### 范围

- Redis 连接和健康状态。
- 注册、登录、聊天、上传按 IP/用户限流。
- 同一会话生成锁。
- 重复请求幂等保护。
- 429 错误结构和前端提示。

### 暂不做

- 不缓存最终医疗回答。
- 不引入复杂消息队列。
- 不把 Session 或业务主数据只存在 Redis。

### 验收

- 超限请求稳定返回 429 和 request_id。
- 不同用户的额度互不影响。
- 同一会话不能并行生成两次。
- 锁有超时且异常后可释放。
- Redis 故障行为有测试和日志。

## 6. 阶段七：RAG 评估与检索优化 `[进行中]`

### 建议拆分任务

1. **任务 7.1 `[进行中]`**：建立版本化的 30-50 题评估集，包含期望文档、关键事实和拒答样例；运行当前系统得到基线，不改检索算法。
2. **任务 7.2**：在不改变回答行为和 API 的前提下，把 `RagService` 拆为查询构造、检索、回答三个小接口。
3. **任务 7.3**：补充科室、主题、文档类型等元数据过滤，并加入可配置最低相关度和拒答策略。
4. **任务 7.4**：增加关键词 + 向量的混合检索，保留只用向量的功能开关和回退路径。
5. **任务 7.5**：通过 `RerankPort` 增加可关闭的 Reranker，不让厂商 SDK 进入 RAG 应用服务。
6. **任务 7.6**：用同一评估集输出质量、延迟和费用对比，删除无收益的复杂度，冻结 `RAG v1.2`。

### 验收

- 来源命中率和拒答准确率可量化。
- 对比报告同时记录质量、耗时和模型费用。
- Reranker 关闭时基础检索仍可运行。
- 检索变化不影响会话持久化和 SSE 协议。
- 评估运行不写入真实用户会话，也不修改知识库生产数据。
- 每项优化都能单独开关和回退，不能同时更换切片、Embedding、召回和 Prompt 后再猜哪项有效。

## 7. 阶段八：可观测性 `[待开始]`

### 建议拆分任务

1. **任务 8.1**：统一 JSON 日志字段和 request_id 上下文，不修改业务结果。
2. **任务 8.2**：记录请求、检索、重排、模型四类阶段耗时和稳定错误类型。
3. **任务 8.3**：记录 Token、估算费用、限流、Redis 降级、中止和失败指标。
4. **任务 8.4**：增加管理员只读统计接口与简单页面，不开放日志全文和敏感正文。
5. **任务 8.5**：补故障注入和保留/轮转策略，冻结 `RAG v1.3`。

### 验收

- 可使用 request_id 串起一次请求。
- 日志不包含密钥、密码、Token 或完整医学正文。
- 能解释一次慢请求耗时在哪个阶段。
- 能查看成功率、平均耗时和主要失败原因。
- 日志和指标通过独立 `TelemetryPort` 写入，业务 Service 不依赖具体日志平台。
- 关闭统计页面或更换监控适配器不影响问答、文档和认证。

## 8. 阶段九：受控资料整理 Agent `[待开始]`

### 开发前提

- 任务 7.2 已提供稳定的 `KnowledgeSearchPort`，Agent 不复制向量检索。
- 任务 8 至少已能记录模型和工具耗时、错误与费用。
- 普通 RAG 完整回归稳定，作为 Agent 开发的不可破坏基线。

### 建议拆分任务

1. **任务 9.1**：建立 `modules/agent` 骨架、Tool Protocol、Tool Registry 和功能开关，不调用真实模型。
2. **任务 9.2**：通过 Alembic 增加 `agent_runs`、`agent_steps`，实现按用户隔离的 Repository 和状态转换测试。
3. **任务 9.3**：实现 `search_knowledge`、`get_document_info` 两个只读工具，只调用公开 Port。
4. **任务 9.4**：实现单 Agent Runtime，加入最大 5 步、单工具超时、Token/费用预算和停止条件。
5. **任务 9.5**：增加 `summarize_document` 与 `generate_learning_report`，复用文档读取和模型 Port。
6. **任务 9.6**：增加独立 Agent REST/SSE 接口、工具事件和安全错误契约，不改普通聊天 SSE。
7. **任务 9.7**：增加独立 `/agent` 页面、运行历史、停止生成和报告展示，不改 `ChatView`。
8. **任务 9.8**：建立 Agent 工具选择、越权、超时、预算、失败恢复评估，冻结 `Agent v2.0`。

### 验收

- Agent 能根据任务选择正确工具。
- 工具只能调用已有 Service，不能直接操作数据库和 Chroma。
- 达到步骤/时间/Token 上限时安全停止。
- 普通 RAG 页面和接口不受 Agent 故障影响。
- 运行与步骤记录按用户隔离，只保存可审计事件，不保存模型隐藏推理过程。
- 第一版不执行系统命令、任意 Python/SQL，不做网页搜索、医疗诊断、多 Agent 或拖拽工作流。

## 11. 每个任务的完成模板

任务只有满足以下条件才能标记完成：

1. 代码只覆盖约定范围，没有顺手堆入下一阶段功能。
2. 新行为有自动化测试，原有回归测试通过。
3. 页面行为相关任务完成一次真实浏览器冒烟。
4. 删除/迁移任务验证目标数据和无关数据。
5. `docs/handoff.md` 记录真实结果和唯一下一任务。
6. 本文档状态更新；架构/接口改变时更新 `technical-design.md`。
7. 明确模型 API 是否被真实调用以及可能产生的费用。
