# 项目想法与产品定位

## 1. 我为什么要做这个项目

我的求职目标是 Python 后端开发或大模型应用开发实习。目前我已经学习过 FastAPI 项目和黑马课程的 RAG/Agent 项目，但原来的 RAG 项目使用 Streamlit，整体更像课程演示，前后端边界、接口设计和工程结构不够突出。

我希望在真实理解原项目代码的基础上，将其改造成一个前后端分离、能够完整展示后端和大模型应用能力的项目。

## 2. 项目主题

项目暂定名：**Medical RAG Assistant（医疗知识库智能问答系统）**。

用户可以上传 PDF/TXT 医疗资料，系统完成文档解析、文本切分、向量化和知识库入库。用户提问后，系统从知识库检索相关片段，再由大模型基于检索内容生成带来源的回答。

项目定位是学习、科研资料查询和内部知识检索，不用于临床诊断。

## 3. 项目从哪里改造

原始项目保存在：

```text
reference/original-rag-agent
```

原项目已经具备：

- Streamlit 聊天页面。
- LangChain Agent。
- 通义千问 `qwen3-max`。
- `text-embedding-v4` Embedding。
- Chroma 向量存储。
- PDF/TXT 加载和文本切分。
- Prompt、YAML 配置、日志和工具调用。

新项目不会简单修改页面名称，也不会把旧代码全部复制过来。改造重点是：

1. 用 FastAPI 提供清晰的 REST API。
2. 用 Vue 3 构建独立前端。
3. 将路由、业务服务、模型、向量存储和配置分层。
4. 返回回答引用来源，提高可追溯性。
5. 增加文档上传和知识库管理能力。
6. 增加统一异常处理，避免网页显示完整 Traceback。
7. 后续增加会话持久化、认证、缓存和部署能力。

## 4. 第一版用户故事

### 知识库管理

作为用户，我可以上传一份 PDF 或 TXT 医学资料。系统能够告诉我文档是否处理成功，并避免重复上传同一个文件。

### 知识库问答

作为用户，我可以针对已上传资料提问。系统应优先依据资料回答，并展示使用了哪些文件或页面。

### 安全拒答

当资料中没有答案时，系统应明确表示“知识库资料不足”，而不是虚构医学结论。

## 5. 第一版不做什么

为了确保自己能够真正理解并完成项目，第一版暂时不做：

- 多智能体协作。
- 复杂工作流编排。
- 微服务拆分。
- 云端向量数据库。
- 医疗诊断、处方或药物剂量推荐。
- 为了丰富简历而堆叠没有实际使用的技术。

## 6. 项目亮点方向

项目完成后，希望能真实展示以下能力：

- 使用 FastAPI 设计和实现后端接口。
- 理解前后端分离的数据交互方式。
- 完成文档解析、切分、Embedding、向量检索和生成回答的 RAG 全链路。
- 使用 Chroma 管理本地向量数据。
- 实现来源引用、知识不足拒答和流式输出。
- 管理 API Key、配置、日志和异常。
- 能够解释一次问题从 Vue 页面到大模型回答的完整数据流。

后续只有在功能真实完成后，才加入 MySQL、Redis、JWT 和 Docker 等简历描述。

## 7. 最终希望达到的效果

用户打开网页后可以：

1. 在知识库页面上传和管理医学资料。
2. 在问答页面输入自然语言问题。
3. 看到逐步生成的答案。
4. 展开查看引用来源和相关原文。
5. 查看和继续历史会话。

对开发者而言，项目应具备清晰目录、可读代码、接口文档、环境配置示例、测试和启动说明，并且可以在面试时独立演示和解释。
