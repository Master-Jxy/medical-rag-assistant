# 项目愿景与产品边界

## 1. 为什么做这个项目

项目作者的求职方向是 Python 后端开发和大模型应用开发。原始课程项目使用 Streamlit，适合学习 RAG 和 Agent，但无法充分展示前后端分离、数据库设计、接口治理、测试和部署能力。

本项目不是简单更换页面，而是把课程 Demo 持续改造成一个**可运行、可解释、可测试、可扩展的医疗知识库应用**。

## 2. 产品定位

Medical RAG Assistant 面向医学资料学习、科研资料整理和内部知识检索场景。

用户可以：

- 注册并登录自己的账号。
- 上传和管理公共 PDF/TXT 医疗资料。
- 对公共知识库提问并查看引用来源。
- 查看只属于自己的历史会话并继续追问。
- 后续使用资料整理 Agent 完成摘要、对比和学习报告。

系统不用于临床诊断，不推荐处方或具体药物剂量，不替代医生判断。

## 3. 当前产品状态

当前 MVP 已完成：

- Vue 3 与 FastAPI 前后端分离。
- PDF/TXT 上传、解析、切分、SHA-256 去重和删除。
- DashScope Embedding + Chroma 向量检索。
- 通义千问回答、来源引用和知识不足拒答。
- SSE 逐块输出、停止生成和安全错误事件。
- MySQL 会话、消息、来源持久化和多轮上下文。
- 会话新建、切换、刷新恢复和删除。
- 后端、前端组件、SSE 字节分片和构建验证。

本地知识库当前登记 27 份较完整的医疗主题文档，共 103 个片段。运行数据不提交 Git。

## 4. 目标用户故事

### 账号与会话

作为注册用户，我登录后只看到自己的历史会话，无法访问其他账号的会话。

### 公共知识库

作为注册用户，我可以检索所有公共医疗资料。新上传资料对所有用户可检索，但只有上传者能够删除；系统内置资料不可由普通用户删除。

### 系统文档管理

作为管理员，我可以新增、删除和整份替换系统文档。普通用户不能把自己提升为管理员，也不能修改系统文档；用户上传资料的审核流程作为后续增强，不影响当前公共上传规则。

### 可靠问答

作为资料使用者，我看到的回答应尽量基于知识库，并能展开查看使用了哪些文件和原文片段。资料不足时系统应明确拒答。

### 资料整理 Agent

作为学习者，我可以让 Agent 选择知识检索、文档摘要、多文档比较或学习报告工具，而不是只进行普通聊天。

## 5. 企业化方向

“企业级”在本项目中不等于堆技术或拆微服务，而是逐步具备：

- 清晰的模块职责和依赖方向。
- 身份认证、数据归属和权限校验。
- 数据迁移、失败回滚和一致性检查。
- 限流、重复请求保护和成本控制。
- 可量化的检索评估与回归测试。
- 结构化日志、请求追踪和运行指标。
- 可控、可审计、有工具上限的 Agent。
- 可重复的 Docker 部署和数据持久化。

### 5.1 对标公开成熟项目的方式

项目会参考 Dify、Langflow、RAGFlow、Coze Studio、OpenHands 等公开项目的工程思路，但不追求复制它们的平台规模：

- 学习 RAGFlow 对文档处理、检索质量和评估的重视。
- 学习 Dify、Langflow、Coze 对工具注册、工作流状态、运行记录和可观测性的拆分。
- 学习 OpenHands 对执行边界、步骤上限、权限和失败恢复的约束。
- 保留当前项目的医疗资料场景、简单部署和可解释数据流。

对标目标不是“功能数量相同”，而是让每个已实现功能都具备清晰边界、可测试性、故障策略和运行证据。

## 6. 明确不做

- 不训练或在本地服务器运行大模型。
- 不提供医疗诊断、处方或治疗决策。
- 不为了简历拆分微服务、引入消息队列或 Kubernetes。
- 第一版 Agent 不做多 Agent 协作、任意代码执行或通用低代码工作流平台。
- 不在没有评估集的情况下宣称检索效果提升。
- 不让 Agent 访问任意系统命令、数据库或未审核工具。
- 不把尚未实现的功能写成已完成功能。

## 7. 成功标准

项目最终应做到：

1. 新开发者只阅读仓库文档即可继续开发。
2. 每个业务模块可以独立测试，修改一个模块不会无故破坏其他模块。
3. 关键流程有自动化测试和真实页面验收。
4. 能从日志和评估结果解释系统为什么回答正确或失败。
5. 能部署到云服务器，由面试官通过网页完成注册、上传、提问和查看引用。
6. 项目作者能够用黑盒数据流解释每个核心功能。
