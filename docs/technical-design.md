# 医疗知识库智能问答系统技术文档

## 1. 文档说明

- 项目名称：医疗知识库智能问答系统
- 项目类型：前后端分离的大模型 RAG 应用
- 当前状态：阶段四已完成并通过后端测试、Vue 组件测试及真实浏览器验收，待整理演示步骤和项目截图
- 目标用户：需要查询医学资料的学习者、科研人员或内部业务人员
- 开发目标：将现有 Streamlit RAG/Agent 项目改造成 Vue + FastAPI 前后端分离项目

> 本项目用于技术学习与作品展示，不提供临床诊断或治疗建议。

## 2. 项目背景

传统关键词搜索只能匹配字面内容，难以理解用户问题的真实语义。通用大模型虽然具备较强的语言理解能力，但可能不了解私有医学资料，也可能生成缺乏依据的回答。

本项目采用 RAG（Retrieval-Augmented Generation，检索增强生成）方案：先从用户上传的医学文档中检索相关内容，再将检索结果与用户问题一同交给大模型生成答案，并返回引用来源，从而提高回答的相关性和可追溯性。

## 3. 项目目标

### 3.1 第一版目标（MVP）

第一版只完成一条完整、稳定的核心链路：

1. 用户上传 PDF 或 TXT 医学资料。
2. 后端解析并切分文档。
3. 调用 Embedding 模型生成文本向量。
4. 将向量和文档元数据保存到 Chroma。
5. 用户在网页中提出问题。
6. 后端检索相关文档片段。
7. 大模型根据检索结果生成回答。
8. 前端展示回答及引用来源。

### 3.2 后续增强目标

- 支持回答流式输出。
- 支持多轮对话及历史会话。
- 增加用户注册、登录和 JWT 认证。
- 使用 MySQL 保存用户、会话、消息和文档信息。
- 使用 Redis 缓存热点问题或会话状态。
- 增加 Agent 工具调用能力。
- 增加 Docker Compose 部署方案。
- 增加接口测试、日志和监控。

## 4. 技术栈

| 层级 | 技术 | 主要用途 |
| --- | --- | --- |
| 前端 | Vue 3 | 构建网页和交互界面 |
| 前端构建 | Vite | 启动和构建 Vue 项目 |
| UI 组件 | Element Plus | 文件上传、表格、消息提示等组件 |
| HTTP 客户端 | Axios | 调用 FastAPI 接口 |
| 后端 | FastAPI | 提供问答、文档管理等 REST API |
| 数据校验 | Pydantic | 校验请求参数和返回数据 |
| ASGI 服务 | Uvicorn | 运行 FastAPI 应用 |
| 大模型编排 | LangChain | 组织模型、Prompt、检索器和 RAG 链路 |
| 聊天模型 | 通义千问 qwen3-max | 理解问题并生成回答 |
| 向量模型 | text-embedding-v4 | 将文本转换为向量 |
| 向量数据库 | Chroma | 保存和检索文档向量 |
| 文档解析 | PyPDF / LangChain Loader | 解析 PDF、TXT 文档 |
| 配置管理 | YAML + 环境变量 | 管理模型名称、路径及 API Key |
| 会话数据库 | MySQL + SQLAlchemy | 保存会话、消息和引用来源 |
| 后续缓存 | Redis | 缓存和会话状态管理 |

## 5. 系统架构

```text
┌─────────────────────────────┐
│         Vue 3 前端           │
│ 问答页面 / 知识库管理页面     │
└──────────────┬──────────────┘
               │ HTTP / SSE
               ▼
┌─────────────────────────────┐
│        FastAPI 后端          │
│ 路由 / 参数校验 / 异常处理    │
└──────────────┬──────────────┘
               │ 调用业务服务
               ▼
┌─────────────────────────────┐
│         RAG 服务层           │
│ 文档处理 / 检索 / Prompt组装  │
└──────────┬──────────┬───────┘
           │          │
           ▼          ▼
┌────────────────┐  ┌────────────────────┐
│ Chroma 向量库   │  │ 阿里云百炼模型服务  │
│ 文档向量与元数据 │  │ Qwen + Embedding    │
└────────────────┘  └────────────────────┘
```

前端不直接调用大模型。所有模型请求均经过 FastAPI 后端，以避免 API Key 暴露在浏览器中。

## 7. 核心业务流程

### 7.1 文档入库流程

```text
用户上传文件
    ↓
FastAPI 检查文件格式和大小
    ↓
保存原始文件并计算文件哈希
    ↓
解析 PDF/TXT 文本
    ↓
按指定长度和重叠范围切分文本
    ↓
调用 text-embedding-v4 生成向量
    ↓
向量、文本、来源和页码写入 Chroma
    ↓
返回处理结果
```

文件哈希用于识别重复文件，避免同一文档被重复向量化和重复计费。

### 7.2 问答流程

```text
用户输入问题
    ↓
FastAPI 校验问题
    ↓
将问题转换为查询向量
    ↓
从 Chroma 检索最相关的文档片段
    ↓
组合系统提示词、文档片段和用户问题
    ↓
调用 qwen3-max
    ↓
返回答案、引用来源和请求标识
```

### 7.3 RAG Prompt 原则

系统提示词应要求模型：

- 优先依据检索到的上下文回答。
- 上下文没有答案时明确说明资料不足。
- 不虚构医学结论、药物剂量或引用来源。
- 将普通健康知识与医疗诊断建议区分开。
- 回答末尾返回所依据的资料来源。

## 8. 第一版接口设计

统一接口前缀：`/api/v1`

### 8.1 健康检查

```http
GET /api/v1/health
```

响应示例：

```json
{
  "status": "ok"
}
```

### 8.2 普通问答

```http
POST /api/v1/chat
Content-Type: application/json
```

请求示例：

```json
{
  "question": "高血压有哪些常见症状？",
  "top_k": 4
}
```

响应示例：

```json
{
  "answer": "根据知识库资料……",
  "sources": [
    {
      "file_name": "高血压诊疗指南.pdf",
      "page": 12,
      "content": "相关原文摘要……"
    }
  ],
  "request_id": "7a6a5f6c-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

### 8.3 上传文档

```http
POST /api/v1/documents
Content-Type: multipart/form-data
```

第一版支持：`.pdf`、`.txt`。

### 8.4 获取文档列表

```http
GET /api/v1/documents
```

### 8.5 删除文档

```http
DELETE /api/v1/documents/{document_id}
```

删除时需要同时删除原始文件、文档记录及其 Chroma 向量。

### 8.6 流式问答（第二阶段）

```http
POST /api/v1/chat/stream
Content-Type: application/json
```

建议使用 SSE（Server-Sent Events），因为该场景主要是服务器单向持续返回文字，实现成本低于 WebSocket。

当前 SSE 事件顺序：

```text
token（可重复） -> sources -> done
```

- `token`：本次生成的一小块文字。
- `sources`：回答使用的引用文件、页码和原文片段。
- `done`：请求 ID 和医疗免责声明。
- `error`：流已经开始后发生的安全错误信息，不包含 Traceback。

## 10. 配置与密钥管理

真实 API Key 只保存在后端环境变量中：

```env
DASHSCOPE_API_KEY=your_key_here
CHAT_MODEL_NAME=qwen3-max
EMBEDDING_MODEL_NAME=text-embedding-v4
CHROMA_PERSIST_DIR=./chroma_db
UPLOAD_DIR=./data
```

要求：

- `.env` 必须写入 `.gitignore`。
- Git 仓库只提交 `.env.example`。
- 前端代码中不能出现 API Key。
- 日志不能打印完整 API Key。

## 11. 异常处理

后端需要统一处理以下问题：

| 场景 | 建议处理 |
| --- | --- |
| 未配置 API Key | 启动时给出明确错误 |
| 余额不足或欠费 | 返回“模型服务暂不可用” |
| 文件格式不支持 | 返回 400 |
| 文件过大 | 返回 413 |
| 文档解析失败 | 保存失败原因并返回 422 |
| Chroma 读写失败 | 记录日志并返回 500 |
| 模型请求超时 | 有限次数重试后返回 503 |
| 知识库无相关内容 | 明确提示资料不足 |

生产接口不应将完整 Python Traceback 返回给前端用户。

## 12. 日志设计

建议记录：

- 请求时间、请求路径和请求 ID。
- 文档上传、解析、切分和向量化状态。
- 检索耗时、模型调用耗时和总耗时。
- 模型名称和 Token 使用量（如果接口返回）。
- 异常类型和必要的上下文。

禁止记录：

- API Key。
- 用户密码。
- 完整敏感医学资料。

## 16. 当前开发决策

1. 复用现有 RAG/Agent 项目的可用代码和经验，不直接在旧项目中大规模修改。
2. 新项目采用独立目录和独立 `.venv`。
3. 第一版先完成普通 RAG，不立即加入 Agent。
4. 第一版先使用本地 Chroma，不引入复杂的云向量数据库。
5. 先通过 FastAPI `/docs` 验证后端，再开发 Vue 页面。
6. 当前继续使用 `qwen3-max` 和 `text-embedding-v4`，后续将模型工厂改为可配置切换。
