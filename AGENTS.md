# Codex Project Guidance

## 1. 项目定位

本项目是用于求职展示和持续学习的医疗知识库智能问答系统。当前已经具备 Vue 3 + FastAPI + MySQL + Chroma 的完整 RAG 主链路、认证与管理员能力，以及 Redis 连接基线；后续按企业应用思路逐步增加接口保护、检索评估、可观测性、资料整理 Agent 和可重复部署。

项目采用**模块化单体**，不是微服务。目标是让每个业务模块职责清晰、可以独立测试和替换，同时保持部署简单，适配当前 2 核 2G 的演示服务器。

> 项目仅用于学习和医学资料检索，不提供临床诊断、处方或治疗建议。

## 2. 新任务开始前的必读顺序

每个新对话必须先阅读：

1. `AGENTS.md`：开发规则和模块边界。
2. `docs/handoff.md`：当前真实状态和唯一下一任务。
3. `docs/development-roadmap.md`：当前阶段的任务与验收标准。
4. `docs/technical-design.md`：数据模型、接口和目标架构。
5. 与当前任务直接相关的源码和测试。

`reference/original-rag-agent/` 只在需要理解旧课程项目时阅读，不允许直接复制或改写其中的数据、密钥和环境。

文档发生冲突时，优先级为：**当前代码与测试 > handoff > roadmap > technical-design > README**。发现冲突必须在本次任务中修正文档。

## 3. 当前阶段

MVP、认证接口、会话用户隔离、文档公共权限、Vue 登录闭环、管理员系统文档管理以及**阶段六 Redis 接口保护**已完成，`RAG v1.1` 已冻结；Docker/阿里云 HTTP 部署基线也已完成，但 HTTPS、备份恢复和完整线上验收尚未结束。当前唯一开发任务仍是：**任务 7.1，建立版本化 RAG 评估集并运行当前系统基线，不修改检索算法**。

当前数据库迁移版本为 `0005_user_role`。会话具有非空 `user_id` 外键，全部会话 CRUD、普通问答和 SSE 问答均按当前用户过滤。文档以 MySQL `documents` 表为主登记源；当前知识库是 27 份系统公共文档、103 个 Chroma 片段，普通用户不能删除，管理员通过独立页面和集中授权接口管理。Vue 已具备完整登录闭环和管理员路由保护。

已经确认的产品决策：

- 使用邮箱和密码开放注册、登录。
- 每个账号只能查看和操作自己的聊天会话。
- 现有测试会话在迁移前备份，迁移时清空。
- 所有知识库文档保持公共可检索。
- 新上传文档也进入公共知识库，但只有上传者可以删除；系统内置资料不可由普通用户删除。
- 基础登录版本使用短期 Bearer JWT，不做邮箱验证码、找回密码和刷新令牌。
- 管理员角色与系统文档管理不改变普通用户现有公共上传规则。
- 管理员系统文档上传跳过小时频率额度，但必须继续复用上传并发租约、文件校验、去重和文档生命周期；禁止在路由或 Redis 适配器里新增角色判断。

Redis 当前为可选基础设施：未配置时健康状态为 `disabled`，连接失败时为 `degraded`；认证、聊天和上传保护使用有界本机兜底，会话生成锁与请求幂等采用 fail-closed，Redis 状态不明确时阻止带会话的昂贵模型调用。健康接口已暴露四类保护能力的策略、模式、成功/失败/恢复计数和最后异常类型，持续故障日志按状态迁移去重且不记录业务正文或密钥。会话 SSE 停止使用可取消的 DashScope 异步 HTTP 流，收到停止信号后取消本地读取任务并关闭流连接，保存 `stopped`、释放生成锁后页面恢复发送。当前只执行任务 7.1；不同时修改检索算法、Agent 或 Docker。

## 4. 模块边界

后端调用方向固定为：

~~~text
API 路由 -> 应用服务 -> Repository / Infrastructure -> MySQL / Chroma / Redis / 模型服务
~~~

必须遵守：

- `api` 只处理参数、身份依赖和响应，不直接写 SQL、Chroma、Redis 或调用模型。
- `service` 负责一个业务用例的编排，不读取 FastAPI Request，不拼接前端展示结构。
- `repository` 只负责 MySQL 查询和持久化，不调用模型或向量库。
- `infrastructure` 封装外部系统，例如 Chroma、DashScope、Redis 和文件存储。
- Pydantic Schema 是 API 契约；SQLAlchemy Model 不直接作为跨模块输入输出。
- RAG 检索和回答生成逐步拆开；会话服务只负责组织流程，不实现检索算法。
- 跨模块复用必须经过小而稳定的 Port/Protocol 或公开应用服务，禁止导入其他模块的 Router、Repository 和私有实现。
- Agent 是独立业务模块。Agent Runtime 只做状态流转和工具调度，工具只能调用公开应用服务，禁止直接查询数据库、操作 Chroma 或访问系统命令。
- 普通 RAG 和 Agent 使用独立路由、用例和 SSE 事件；Agent 故障不得改变现有聊天接口契约。
- Redis 通过限流和锁适配器提供能力，业务模块不接触 redis-py 客户端，也不能把业务主数据只放在 Redis。
- 通用代码只放配置、异常、日志、安全和请求标识；不要建立包含业务逻辑的“万能工具类”。

禁止出现：

- 路由文件里出现复杂业务判断。
- Vue 页面直接调用模型、MySQL、Chroma 或拼接密钥。
- 一个服务同时负责认证、文档、会话、检索和 Agent。
- Agent 工具绕过应用服务，直接调用 Repository、SQLAlchemy Session 或第三方 SDK。
- 为了模仿高星项目一次引入多 Agent、可视化工作流、消息队列和 Kubernetes。
- 为了复用少量代码建立跨模块循环依赖。
- 未完成的功能提前写进简历或 README 的“已实现功能”。

## 5. 增量重构原则

- 不做一次性大搬家。开发某个功能时，只重构该功能触及的模块。
- 移动代码前先补或确认回归测试，移动后保持接口行为不变。
- 新模块通过构造参数或 FastAPI 依赖注入获得依赖，避免在业务代码中到处创建全局客户端。
- 共享接口优先使用小而稳定的 Protocol 或服务方法，不暴露第三方库对象。
- 任何数据库结构变化都通过 Alembic 迁移，不手工修改生产表。
- 涉及 MySQL、文件和 Chroma 的组合写入必须设计失败回滚和一致性验证。
- 跨 MySQL、文件和 Chroma 不存在真正的原子事务，文档写操作使用共享生命周期服务和可补偿流程，不复制多套实现。
- 管理员授权只通过集中的 `require_admin` 或权限策略完成；禁止在路由、页面和多个 Service 中散落角色判断。
- JWT 只识别用户，数据库角色才是授权真相源；禁止信任前端传入的角色或仅靠隐藏按钮保护接口。
- 管理员模块不得修改会话、RAG Prompt、SSE 协议或普通用户隔离逻辑。
- 新功能必须先写清输入、输出、失败策略和所属模块，再开始编码；没有模块归属的功能不得直接塞进现有 Service。
- 任何大任务先拆成单模块、可测试、可回退的小任务；每次只允许一个主模块发生行为变化。

## 6. 安全和成本约束

- API Key、JWT 密钥和数据库密码只从环境变量或本地 `.env` 读取。
- `.env`、上传文件、Chroma 数据、日志和数据库备份不得提交 Git。
- 密码只能保存安全哈希，禁止明文密码进入数据库、日志和异常。
- 日志默认不记录完整问题、回答、Prompt、Token 或医学文档正文。
- 测试优先使用假模型和假 Embedding；只有明确需要真实验证时才调用 DashScope。
- 禁止无限重试、循环模型调用和没有上限的 Agent 工具循环。

## 7. 验证要求

修改范围决定验证范围，但每阶段至少包括：

~~~powershell
# 后端
backend\.venv\Scripts\python.exe -m pytest -q backend/tests

# 前端组件、SSE 和构建
D:\Nodejs\npm.cmd --prefix frontend test
D:\Nodejs\npm.cmd --prefix frontend run test:stream
D:\Nodejs\npm.cmd --prefix frontend run build
~~~

涉及页面交互时，还要进行一次真实浏览器验收；涉及数据删除或迁移时，必须验证目标数据被处理且无关数据未受影响。

## 8. 每次任务的交付格式

每次完成一个任务后必须：

1. 说明改了什么，以及没有改什么。
2. 用黑盒数据流解释功能如何工作。
3. 列出执行过的测试和结果。
4. 更新 `docs/handoff.md` 的当前状态与下一任务。
5. 更新 `docs/development-roadmap.md` 对应任务状态。
6. 若接口、数据模型或模块边界变化，同步更新 `docs/technical-design.md`。

一次只留下一个明确的“下一任务”，让新的开发窗口无需依赖旧聊天记录也能继续。
